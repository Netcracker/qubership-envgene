# Criteria for SBOM generation and versioning

This section describes when EnvGene (re)generates Application SBOMs and how the SBOM generator version is used.

## SBOM generator version

- Each Application SBOM generated by the library must include in the `tools.components[0]` section:
  - `name`: always `sbom_generator`
  - `version`: the version of the SBOM generator (library) that produced this SBOM

- The SBOM generator version must be increased **only** when the SBOM **specification** (contract) changes, for example:
  - the structure or semantics of the generated SBOM change
  - new mandatory attributes are introduced

- Changes that do not affect the SBOM contract (e.g. performance improvements, refactoring, non-breaking bug fixes) must **not** change the generator version.

## Per-application regeneration logic (EnvGene)

EnvGene must decide for **each** application in the Solution Descriptor independently whether to generate or skip SBOM. Processing each application (each `app:ver`) is independent.

- **File name pattern** for Application SBOM: `<application>:<version>.sbom.json`
  - Paths: `/configuration/sboms/<application>:<version>.sbom.json` and `/environments/<ENV_NAME>/Inventory/sboms/<application>:<version>.sbom.json`

- For each application:
  1. **If the SBOM file exists:**
     - Read `tools.components[0].version` from the existing SBOM.
     - Compare it with the current SBOM generator version.
     - If the versions **differ** — re-generate the SBOM **only for this application** and overwrite the file.
     - If the versions **match** — do **not** re-generate.
  2. **If the SBOM file does not exist** — generate the SBOM for this application with the current generator version and persist it.

- The previous behaviour (re-generate **all** SBOMs when at least one was missing) is **deprecated** and must be replaced by this per-application logic.
